## 2. Chi tiết từng vấn đề đã sửa

### Nhóm A — Chất lượng retrieval (recall/ranking)

#### V1 — Calibrated fusion (chuẩn hóa thang điểm)
- **Vấn đề:** text-signal (RRF ∈ [0, 0.016]) bị review_score (∈ [0,1]) nuốt ~24× khi cộng thẳng → FULL ≈ NO-SERVICE.
- **Sửa:** `_minmax_norm` chuẩn hóa text-signal về [0,1] trước fuse trong `business_rerank` (port `_minmax` từ develop-2 `calibrated_rrf_fusion`).
- **Phát hiện khi đo:** chuẩn hóa với trọng số cũ `neural=0.5` làm recall **tụt** (0.52→0.44) vì text retrieval **yếu hơn nhãn KE** trên corpus này. Sweep trọng số → chốt **`neural=0.05`** (text-signal đóng vai trò tinh chỉnh, không áp đảo).
- **File:** [retrieval/reranking/fusion.py](../../../retrieval/reranking/fusion.py)
- **Kết quả:** MRR 0.9042→0.9065, Hit 0.9661→0.9831.

#### V9 — RRF hợp nhất ở cấp hotel
- **Vấn đề:** BM25 trả `chunk_id="bm25_hotel_<id>"` (document-level), Qdrant trả chunk_id thật (chunk-level) → RRF dedupe theo chunk_id **không bao giờ trùng** → không cộng dồn điểm 2 nguồn.
- **Sửa:** `rrf_by_hotel` — gom mỗi nguồn về hotel_id (rank tốt nhất), RRF trên hotel_id. (Chọn cách của dự án, an toàn hơn reindex chunk-level của develop-2.)
- **File:** [retrieval/reranking/fusion.py](../../../retrieval/reranking/fusion.py), nối trong [pipeline.py](../../../retrieval/hybrid_search/pipeline.py)

#### V5 — IDF concept weighting (ĐÒN BẨY RECALL LỚN NHẤT)
- **Vấn đề:** `build_candidates` sort thuần review_score. Concept lookup coi mọi concept như nhau, nhưng `STYLE_LIVELY` = 1/520 hotel (đặc trưng) vs `OBJ_HOTEL` = 393/520 (vô nghĩa). Hotel khớp concept hiếm bị hotel review-cao chen mất khỏi cap → recall=0 cho câu cảm xúc.
- **Sửa:** thêm **IDF weighting** (`idf = log(N/df)`) trong concept_index; `build_candidates` sort theo `(idf_score, match_count, review_score)`.
- **File:** [retrieval/filtering/concept_index.py](../../../retrieval/filtering/concept_index.py), [retrieval/filtering/hard_filter.py](../../../retrieval/filtering/hard_filter.py)
- **Kết quả:** Recall@10 0.5186→**0.5495 (+3.1%)**, Hit@10→**1.0**, Recall@50→0.95. Hotel GS-015 từ hạng candidate 205 → #1.
- **Ghi chú:** đây là giải pháp **thuần dự án** — develop-2 không có.

#### V3 — Fallback khi candidate rỗng
- **Vấn đề:** query không khớp city/concept → candidate rỗng → màn hình trắng.
- **Sửa:** fallback 2 tầng trong pipeline (vector broad search → review-top).
- **File:** [retrieval/hybrid_search/pipeline.py](../../../retrieval/hybrid_search/pipeline.py)
- **Ghi chú:** không đổi golden (golden có 0 câu rỗng) — là robustness cho query tự do của user.

### Nhóm B — Latency & Index

#### V11 — Cắt latency
- **Sửa 1:** `qdrant_service` `with_payload=True` → chỉ 6 field cần (trước kéo 60+ field).
- **Sửa 2:** `inmemory_hard_filter` cache `_city_blobs` (lru) — trước normalize 520 hotel mỗi request.
- **File:** [retrieval/vector_search/qdrant_service.py](../../../retrieval/vector_search/qdrant_service.py), [retrieval/filtering/hard_filter.py](../../../retrieval/filtering/hard_filter.py)
- **Kết quả:** hard_filter 227ms→**2ms**, Qdrant payload nhẹ −90%. E2E ~1900→1660ms.
- **Phát hiện:** nút cổ chai thật còn lại = **embed bge-m3 trên CPU (~700ms)**, không phải Qdrant. Trần phần cứng — cần GPU.

#### V10 — Analyzer tiếng Việt cho BM25
- **Vấn đề:** `analyzer=standard` không hiểu tiếng Việt (không match từ ghép, không xử lý dấu).
- **Sửa:** `vietnamese_analyzer` (port từ develop-2: standard tokenizer + lowercase + decimal_digit + asciifolding `preserve_original`) → sinh cả token có dấu lẫn không dấu → match query gõ thiếu dấu.
- **File:** [indexing/bm25_index/index_mapping.json](../../../indexing/bm25_index/index_mapping.json), reindex sang `vsf_hotels_bm25_vi` qua [reindex_vietnamese.py](../../../indexing/bm25_index/reindex_vietnamese.py)
- **Kết quả:** Recall@10 0.5114→0.5186 (+0.7% trên golden gõ chuẩn dấu; lợi ích thật lớn hơn cho user gõ thiếu dấu).

#### V14 — Qdrant optimize
- **Vấn đề:** indexed_vectors 12556 < points 13838 (9% chưa build HNSW), status grey.
- **Sửa:** `PATCH optimizers_config.indexing_threshold=100` → build hết.
- **Kết quả:** status grey→**green**, indexed=points (100%).

#### V15 — Embedding CUDA path
- **Sửa:** thêm nhánh `cuda` + env `EMBEDDING_DEVICE` (trước chỉ check mps→cpu).
- **File:** [indexing/embedding/models.py](../../../indexing/embedding/models.py)
- **Ghi chú trung thực:** máy hiện tại CPU-only (`torch.cuda.is_available()=False`) → fix này chỉ có tác dụng khi deploy có GPU.

#### V16 — BM25 source fields
- **Vấn đề:** `DEFAULT_SOURCE_FIELDS` thiếu `description` (mà `_map_hit` đọc) → GET /search trả `description: null`.
- **Sửa:** bổ sung description/amenities/images/source_url/review_count/lat-long.
- **File:** [retrieval/lexical_search/service.py](../../../retrieval/lexical_search/service.py)

## 4. Sự cố hạ tầng đã xử lý trong phiên

- **Ổ C đầy 95%** chặn reindex (OpenSearch flood-stage watermark). Hóa ra **không phải Docker** (Docker ở ổ D, vhdx 0.1GB) mà do cache trên C (`.cache/huggingface` 9.6GB, npm/pip, `.ollama` 4.4GB). Dọn npm+pip+ollama ~8.5GB → C về 90% → reindex chạy được. **Index thật không bị động chạm.**
- **Server API chạy code cũ:** phát hiện 2 process uvicorn khởi động từ 17/06 (trước mọi fix) tranh cổng 8000 → frontend lỗi. Kill + restart 1 server sạch từ code mới. E2E test lại: /search, /context (có evidence V6), /hybrid_search, /ui đều 200.
